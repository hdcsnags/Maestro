# Maestro Build v2 — Task-Queued Execution

*This spec replaces lane-sized generation with task-queued execution. Every agent working on Build v2 must read this before writing code.*

*Refined with input from OpenAI GPT-5.4 (extended thinking) and Claude Opus 4.6 (CLI agent). All six refinements accepted and integrated.*

---

## Core Principles

1. **Pre-Build chooses the cast. Build never re-casts.** What Pre-Build locks is law. The task queue uses only that cast. No agent is added, removed, or swapped during execution unless an explicit reroute occurs for a failed task.
2. **One file in, one file out.** Every orchestrate call produces exactly one file. No multi-file manifests, no giant JSON blobs, no praying it finishes.
3. **Build tasks are NOT rounds.** Council/chat rounds stay `rounds`. Build tasks live in `build_tasks`. Do not overload the round/response data model with per-file build work.
4. **Frontend executes first, worker executes later.** v2.0 runs the execution loop from a frontend hook. v2.5+ should allow background/job execution without rewriting the contract.
5. **Continuation is an escape hatch, not the normal path.** With one-file-per-task, continuation should almost never trigger. If a single file needs continuation, the task decomposition was wrong.
6. **Failures are first-class data.** Every task carries `skip_reason`, `failure_reason`, `provider_error`, and `rerouted_from` so the UI truth layer and debugging are always possible.

---

## The Problem Build v1 Had

Build v1 asked each builder agent to generate an entire lane of files in one response. This meant:

- A 12-file lane required the LLM to produce all 12 complete files as a single JSON blob.
- Supabase edge functions have a ~26-second timeout.
- Large responses routinely 504'd or truncated.
- The "continuation" mechanism was a band-aid — it tried to resume a failed giant response instead of preventing the failure.
- The UI had no meaningful progress because nothing completed until everything completed.
- Failures killed the entire lane instead of one file.

The pattern was: plan big → generate big → pray it finishes.

## What Build v2 Does Instead

Build v2 breaks every lane into individual **build tasks**, one file per task. Concierge plans the queue. The execution hook dispatches tasks one at a time per builder. The UI shows real-time progress per task. Failures reroute to fallback builders. Nothing depends on a single giant response.

The pattern is: plan granularly → execute incrementally → track everything → reroute failures → finish the job.

## Architecture

### Phase 1 — Pre-Build (already exists, extended)

Pre-Build outputs a **build contract** that includes everything it already produces, plus:

```
build_contract = {
  // Existing
  locked_builder_roster: AgentId[],
  lane_assignments: { agent_id, lane_paths }[],
  architect_md: string,
  build_spec_locked: true,
  
  // New in v2
  task_queue: BuildTask[],
  fallback_candidates: { agent_id, lane_paths }[],
}
```

The task queue is generated by Concierge by decomposing each lane's file list into individual tasks:

```typescript
interface BuildTask {
  task_id: string;              // e.g., "task-001"
  file_path: string;            // e.g., "src/api/routes/search.ts"
  lane_owner: AgentId;          // primary builder for this file
  fallback_owner: AgentId | null; // reroute target if primary fails
  dependencies: string[];       // task_ids that must complete first (usually empty)
  status: 'queued' | 'dispatched' | 'waiting' | 'completed' | 'failed' | 'rerouted' | 'skipped';
  retry_count: number;
  max_retries: number;          // default 2
  prompt_slice: string;         // file-specific build instruction derived from architect_md

  // Failure/reroute metadata (OpenAI refinement #5)
  skip_reason: string | null;
  failure_reason: string | null;
  provider_error: string | null;
  rerouted_from: AgentId | null;

  // Result (populated on completion)
  result: null | {
    content: string;
    operation: 'create' | 'update' | 'delete';
    builder_agent: AgentId;
    completed_at: string;
  };
}
```

### Phase 2 — Build Execution Loop

The execution loop runs in a dedicated frontend hook (`useBuildExecution.ts`). It does NOT depend on a single edge function call completing the entire build.

> **v2.0**: Frontend hook owns the loop. If the user closes the tab, the build pauses — but all completed tasks persist in `build_tasks` and can resume on return.
>
> **v2.5+**: The loop can move to a backend worker (Supabase pg_cron, Deno Deploy worker, or similar) that pulls from the persisted task queue. The contract (task statuses in DB, results in DB) stays identical — only the executor location changes.

```
while (queue has tasks with status === 'queued') {
  // 1. Pick next eligible tasks (no unmet dependencies)
  eligible = queue.filter(t => t.status === 'queued' && dependencies_met(t))
  
  // 2. Dispatch up to N tasks in parallel (N = number of active builders)
  //    Each builder gets at most 1 task at a time to avoid overloading providers
  batch = pick_one_per_builder(eligible)
  
  for each task in batch {
    task.status = 'dispatched'
    persist(task)  // write to build_tasks table
    updateUI(task)
    
    // 3. Call orchestrate for ONE file
    dispatch(task.lane_owner, {
      prompt: task.prompt_slice,
      file_path: task.file_path,
      mode: 'build_task',  // signals orchestrate to use the lightweight single-file path
    })
  }
  
  // 4. Collect results as they arrive
  for each completed response {
    if (success && valid_content) {
      task.status = 'completed'
      task.result = { content, operation, builder_agent, completed_at }
      persist(task)
      updateUI(task)
    }
    else if (retryable && task.retry_count < task.max_retries) {
      task.retry_count++
      task.failure_reason = error_message
      task.provider_error = provider_raw_error
      task.status = 'queued'  // re-enter queue
      persist(task)
      updateUI(task)
    }
    else if (fallback_owner exists) {
      task.rerouted_from = task.lane_owner
      task.lane_owner = task.fallback_owner
      task.fallback_owner = null
      task.failure_reason = error_message
      task.status = 'queued'  // re-enter queue with new owner
      persist(task)
      updateUI(task)
    }
    else {
      task.status = 'failed'
      task.failure_reason = error_message
      task.provider_error = provider_raw_error
      persist(task)
      updateUI(task)
      // Surface to user: "This file could not be generated. Skip or retry manually."
    }
  }
}

// 5. All tasks resolved → collect completed results → Execute to GitHub
```

### Phase 3 — GitHub Execution

Same as today. Collect all `task.result` entries with `status === 'completed'`, assemble into the patches array, call `github-execute`. No change to the commit/PR/backup-branch pipeline.

### Phase 4 — Bouncer

Same as today. Bouncer reviews the committed output. No change needed.

## UI States Per Task

The user sees a task board where each file shows its current state:

| State | What the user sees |
|-------|-------------------|
| `queued` | File name, dimmed, waiting in line |
| `dispatched` | File name, pulsing/active indicator, "Sending to [Agent Name]..." |
| `waiting` | File name, spinner, "Waiting on [Provider]..." |
| `completed` | File name, green check, expandable to preview content |
| `failed` | File name, red X, failure reason shown, "Retry / Skip / Reroute" actions |
| `rerouted` | File name, orange arrow, "Rerouted from [Original] to [Fallback]..." |
| `skipped` | File name, grey dash, skip reason shown |

Progress bar at the top: `X of Y tasks completed`.

This is the "momentum" UX that Bolt gets right. The user sees files completing one by one in real time. Not a spinner followed by a wall of JSON.

## What Changes From v1

| Area | v1 | v2 |
|------|----|----|
| Build unit | Entire lane (many files) | One file per task |
| Orchestrate call | One call per lane, huge response | One call per file, small response |
| Timeout risk | High (26s wall for 12+ files) | Near zero (one file = 3-8s) |
| Failure blast radius | Entire lane lost | One file, retryable |
| Reroute capability | None | Per-task fallback to different builder |
| UI progress | None until lane completes | Real-time per-file updates |
| Continuation | Band-aid retry of giant response | Escape hatch only — for oversized single files |
| Parallelism | One builder per lane, sequential | One task per builder, multiple builders simultaneous |
| Data model | Overloaded rounds/responses | Dedicated `build_tasks` table |

## What Does NOT Change

- Pre-Build phase flow (intake, architect, spec freeze, lane assignment) — extended, not replaced.
- GitHub execute pipeline — same patches format, same commit/PR/backup flow.
- Bouncer — same review, runs after execution.
- Concierge — still the orchestration brain. Now it also generates the task queue.
- ARCHITECT.md — still the build source of truth. Task prompt slices are derived from it.
- Lane scope enforcement — still governed by `build_lanes.lane_paths`.

## What DOES Change in Orchestrate

Orchestrate is NOT rewritten, but needs **targeted changes** (OpenAI refinement #2):

1. **Single-file build prompt path** — when `mode === 'build_task'`, use a lighter prompt template. Don't inject the full ARCHITECT.md system prompt for a 30-line utility file. Instead: project summary + file-specific instruction + format constraint.
2. **Per-task timeout behavior** — a config file ≠ a 50-route API module. The execution hook can set `max_tokens` per task based on expected file complexity.
3. **Provider error normalization** — return structured errors (`{ retryable: boolean, error_code, message }`) so the execution loop can distinguish "retryable 504" from "permanent auth failure" from "content policy block."

## Prompt Slice Generation

Each task needs a `prompt_slice` — a focused instruction for generating exactly one file. This is derived from ARCHITECT.md during task decomposition in Pre-Build.

Example: if ARCHITECT.md says lane "backend-api" owns `src/api/routes/search.ts`, `src/api/routes/profile.ts`, `src/api/routes/settings.ts`, then three tasks are created with prompt slices like:

```
"You are building file: src/api/routes/search.ts
This file implements the search route for the CTF training lab.
It should expose a GET / endpoint that accepts a query parameter `q` and performs a SQL search against the orders table.
This is the VULNERABLE version — use string concatenation, not parameterized queries.
Return ONLY a JSON object: { "path": "src/api/routes/search.ts", "content": "<full file content>", "operation": "create" }
Do not include any other files. Do not include explanation outside the JSON."
```

The key constraint: one file in, one file out. The prompt tells the agent exactly what to build and exactly what format to return. No ambiguity, no multi-file manifests.

Prompt slice quality is the make-or-break for Build v2. The concierge model for task decomposition must be Opus-class or GPT-5.4-class. Haiku is fine for triage and chat. Build planning needs a model that can read ARCHITECT.md (which varies wildly in quality depending on which model generated it) and produce correct, ordered tasks with meaningful per-file instructions.

---

## Concierge Role in Build v2

Concierge's build-planning responsibility expands:

1. **Task decomposition** — break architect lane assignments into individual file tasks.
2. **Dependency detection** — if file B imports from file A, task B depends on task A. (Simple heuristic: config/types files first, then lib, then routes/components, then entry points.)
3. **Prompt slice generation** — write the per-file instruction for each task, derived from ARCHITECT.md.
4. **Fallback assignment** — for each task, identify a fallback builder from a different provider in case the primary fails or times out.
5. **Builder count recommendation** — based on task count and project complexity, recommend how many builders to activate.

## Database Changes

### New table: `build_tasks`

```sql
create table if not exists build_tasks (
  id uuid primary key default gen_random_uuid(),
  session_id uuid references sessions(id) not null,
  build_round_id uuid references rounds(id),
  task_id text not null,
  file_path text not null,
  lane_owner uuid references agents(id),
  fallback_owner uuid references agents(id),
  dependencies text[] default '{}',
  status text not null default 'queued',
  retry_count int default 0,
  max_retries int default 2,
  prompt_slice text,

  -- Failure/reroute metadata
  skip_reason text,
  failure_reason text,
  provider_error text,
  rerouted_from uuid references agents(id),

  -- Result (populated on completion)
  result_content text,
  result_operation text,
  result_builder uuid references agents(id),
  completed_at timestamptz,

  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

-- RLS
alter table build_tasks enable row level security;

create policy "Users can manage build tasks in their workspace sessions"
  on build_tasks for all using (
    session_id in (
      select s.id from sessions s
      join workspaces w on s.workspace_id = w.id
      where w.user_id = auth.uid()
    )
  );

-- Indexes
create index idx_build_tasks_session_status on build_tasks(session_id, status);
create index idx_build_tasks_session_id on build_tasks(session_id);
```

### Existing tables — clarification

- `build_lanes` — still used for scope enforcement. Tasks inherit lane scope from their `lane_owner`'s lane assignment.
- `rounds` / `responses` — **NOT used for build tasks.** Council/chat rounds stay rounds. Build tasks are their own data model. An optional `build_round_id` can group a wave/batch for audit purposes, but one file task is not a council round.
- `execution_runs` — still used for GitHub execution. Populated from completed tasks.

## What This Enables Later

Once Build v2 is working:

- **Backend execution (v2.5)** — move the task loop to a Supabase worker / Deno Deploy job. Frontend observes via DB subscriptions. Same contract, different executor.
- **Streaming progress via Supabase Realtime** — subscribe to `build_tasks` status changes instead of polling. Real SSE-style updates.
- **Parallel multi-builder execution** — dispatch tasks to different builders simultaneously with no timeout risk.
- **Smart rerouting** — if Anthropic is slow, reroute queued tasks to OpenAI mid-flight.
- **Build resume** — if the user closes the browser, tasks persist in DB. Reopen → resume from where you left off.
- **Build cost estimation** — task count × average tokens per task = estimated API cost before execution.
- **Incremental builds** — only re-run tasks for files that changed in the spec. Don't rebuild the whole project.

This is the foundation for everything Maestro needs to beat Bolt and then go further.
